# LinkedIn Post Draft — 2026-03-30

## Draft 1: The .claude folder is infrastructure

I've been building an AI-native SDLC for the past month. Today I hit an inflection point.

I had seven engineering standards — testing, security, branching, diagnosis, usability, commits, deployment. Well-written. Thorough. Completely useless to the AI agents actually building the software.

The standards existed as documentation. The agents needed them as operating instructions.

So I repackaged everything: compressed each standard into a path-scoped rule that loads only when the agent touches relevant files. Testing rules load when you touch .ts files. Usability rules load when you touch page components. Security rules load when you touch config or Dockerfiles.

Then I added slash commands that encode the workflows — /diagnose runs the three-step root cause analysis, /spec generates structured specs from product canvases, /chronicle writes session records.

Three named agent personas with restricted tool sets. A code reviewer that can only read. A spec writer that can read and write but not execute. A session chronicler.

One skill that auto-triggers when the agent tries to start coding: "Did you do the canvas? The spec? The design? Are you on a branch?" If not, it blocks.

The insight came from studying three practitioners' setups (Barry Hurd, Eduardo Ordax, Boris Cherny). They all reached the same conclusion from different angles:

Your standards are only as good as your delivery mechanism. Documentation that agents don't load is documentation that doesn't exist.

The .claude folder is Claude's operating system for your project. Get it right and the agent stops guessing.

## Draft 2: The repo is the state handoff

I work on my platform from three devices — Mac, mobile, cloud sessions. Different machines, different contexts, same codebase.

The solution isn't syncing conversation history. It's making the repo the single source of truth for what happened and what to do next.

Every session writes a chronicle entry (what happened, decisions made, open threads). Every session updates memory. Every session leaves SDLC trace logs.

When a new session starts on any device, it does: git pull → read memory → check for missing chronicles → read open issues → propose work → wait for confirmation.

No conversation history needed. No "continue from where I left off." The repo IS the handoff.

The hardest part? Parallel agent safety. What happens when I accidentally start a session on my phone while my Mac is still running one? Session locks, worktree isolation, stale lock detection. Simple protocol, prevents lost work.

## Draft 3: I audited my own system

Today I audited the .claude/ setup across two repos. One had 30 files: 8 rules, 13 skills, 9 hooks. The other had... 0.

The first repo (POA) has been my proving ground for a month. Every lesson learned became a hook, a rule, or a skill. Temperance (pause before implementing). Diagnosis (understand before fixing). Verification (test per task, not batched).

The second repo (platform-docs) had great standards. None of them were wired into the agent.

The gap isn't content. It's delivery. Your standards exist as prose? The agent will read them if you're lucky. Your standards exist as rules with YAML frontmatter that load contextually? The agent follows them every time.

20 files later, both repos are wired. The next step: --add-dir so the app repo inherits the platform standards without duplicating them.

One source of truth. Multiple consumers. Infrastructure, not documentation.

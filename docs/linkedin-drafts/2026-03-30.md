# LinkedIn Post Drafts — 2026-03-30

## Draft 1: The substrate (recommended — strongest hook)

I spent today building what I'm calling the "substrate" — the .claude/ infrastructure that turns engineering standards into an agent's operating system.

Here's what I mean.

I had 7 engineering standards. Testing. Security. Branching. Diagnosis. Usability. Commits. Deployment. Well-written. Thorough. Completely invisible to the AI agents building the software.

The standards existed as documentation. The agents needed them as operating instructions.

So I repackaged everything:

→ 9 rules with path-scoped loading. Testing rules load when you touch .ts files. Usability rules load when you touch page components. Security loads on Dockerfiles. The agent never sees rules that don't apply to what it's touching.

→ 14 skills that encode workflows. /diagnose runs root cause analysis (Is/Is Not → Five Whys → Hypothesis). /temperance forces a pause before building ("Is this the simplest correct approach? Am I brute-forcing?"). /verify gates every commit by change type.

→ 7 hooks that fire automatically. Session starts with mandatory reflection. Tool errors trigger diagnosis before retry. Code edits require SDLC evidence. Commits require verification proof.

→ 3 agent personas with restricted tools. A code reviewer that can only read. A spec writer that can read and write but not execute. Safety through capability restriction, not trust.

The insight came from studying three practitioners — Barry Hurd (specification quality is the bottleneck, not code generation), Eduardo Ordax (.claude is Claude's OS), and Boris Cherny at Anthropic (worktrees for parallel agents, --add-dir for cross-repo standards).

They all reached the same conclusion from different angles:

Your standards are only as good as your delivery mechanism. Documentation that agents don't load is documentation that doesn't exist.

I call the infrastructure layer the "substrate." The UX built on top — what lets anyone from kids to advertisers use it — is the "HUD."

Today was substrate day. 33 files. 5 commits. The foundation for autonomous multi-agent development.

## Draft 2: The repo is the state handoff (session continuity angle)

I work from three devices. Mac. Phone. Cloud. The constraint isn't which machine — it's that context dies between sessions.

The fix isn't syncing conversation history. It's making the repo the single source of truth.

Every session writes:
- A chronicle entry (what happened, decisions, open threads)
- Memory updates (what was learned)
- SDLC trace logs (which quality gates fired)
- LinkedIn drafts (if something shipped worth sharing)
- Danger mode summaries (if the agent ran unsupervised)

When a new session starts — any device — hooks fire automatically:
- Detect if another session crashed (dirty working tree? stale processes?)
- Check for missing chronicles
- Check if another agent is already active (session lock)
- Show recent commits and open issues
- Force the agent to reflect before acting

`git pull` is the onboarding. The repo IS the handoff.

The hardest part? Parallel agent safety. Two sessions on the same branch = lost work. Session locks, worktree isolation, stale lock detection. Simple protocol, critical protection.

## Draft 3: I audited my own system (self-awareness angle)

I audited the .claude/ setup across two repos today.

One had 30 files: 8 rules, 13 skills, 9 hooks. Automated session start/end. Diagnosis chains. Pre-build gates. Post-commit verification. Schema drift detection.

The other had... 0.

Same author. Same standards. Same principles documented. But one repo had the infrastructure to enforce them, and the other had documentation nobody reads.

The gap isn't content. It's delivery.

So I promoted everything that was general enough to be a standard. Stripped the app-specific details. Generalized the hooks. Packaged it all into what I'm calling the "substrate" — the shared .claude/ infrastructure that any repo can inherit.

The result: 9 rules, 14 skills, 7 hooks, 3 agents. One command (`--add-dir`) to inherit all of it.

Now both repos run the same playbook. And the next repo will too.

## Draft 4: The temperance skill (shortest, most shareable)

I added a skill called /temperance to my AI development setup.

It does one thing: forces the agent to pause before building.

Five questions:
1. Is this the simplest correct approach?
2. Am I brute-forcing instead of understanding?
3. What's the blast radius?
4. Can I verify this works in under 2 minutes?
5. If this is wrong, how hard is it to undo?

It exists because of one session where 8 tasks were batch-built without verification. Felt productive. Created more rework than it saved.

The hardest part of AI-assisted development isn't getting the agent to write code. It's getting it to stop and think first.
